PREDICT · RLVR on MBPP · commit 9eefac7

Reactive vs. predictive
coding agents, matched and RLVR'd.

How do reactive and predictive multi-turn coding-agent training systems compare under matched tasks, models, budgets, and rewards? Arm A uses SFT and RLVR for test-and-recover behavior; Arm B adds outcome prediction with verified-label cross-entropy and a keep-or-revise decision before execution.

n=500 held-out test 2×2 seed matrix, both arms Qwen3-4B-Base 8 tool calls / rollout, 30s timeout McNemar + paired bootstrap

Architecture

How the training loop actually runs

PRIME-RL's orchestrator dispatches each rollout into a fresh Verifiers subprocess sandbox, scores it, forms GRPO groups, and sends batches to the trainer — which broadcasts updated policy weights straight back to the inference server every step.

Both arms share one harness. Arm A reacts to test output; Arm B must predict the outcome and commit to KEEP or REVISE before it's allowed to see the real result.

Data

MBPP, split once, touched once

MBPP (Mostly Basic Python Problems): ~1,000 short, crowd-sourced Python problems with gold-standard test assertions, from Austin et al., “Program Synthesis with Large Language Models” (2021). The RLVR-on-MBPP-with-small-Qwen setup was directly inspired by Skopin & Kotelnikov, “Improving Small Language Models for Code Generation with Reinforcement Learning from Verification Feedback” (2026).

Scope
Arm A is a matched comparator, not a single-variable ablation. Arm B bundles the predict/decide protocol, an added action space, the auxiliary CE loss, and a different SFT trace format together — this compares two complete systems, it does not isolate “prediction” as one causal factor. Diagnosis

What was actually hindering Arm B

Decision-following is solid — KEEP follows a PASS prediction 100% of the time, REVISE follows a non-PASS prediction 97–99% of the time, in both seeds. That's not the problem. The problem is prediction coverage, by outcome class, step 100 seed 42:

Recall = correct predictions / actual occurrences of that class. The model has not once correctly predicted ASSERTION_FAILURE — the dominant failure mode — at step 50, 75, or 100, in either independent seed.

Blind spot
100% PASS at SFT (fully collapsed) → a brief 1–2% ASSERTION_FAILURE recall at step 25 → extinguished back to 0% by step 50, never recovering. Correctly predicting ASSERTION_FAILURE and choosing REVISE saves at most one turn over just testing and reacting — final task reward is identical either path, so GRPO has no gradient toward that discrimination skill. Only the auxiliary CE loss (λ=0.1) could teach it, and it isn't strong enough to survive against GRPO's pull.

What this shows
Arm B predicts PASS and KEEPs three times in a row on a candidate that hasn't run yet — matching the 0% ASSERTION_FAILURE recall above — then reacts to the real test failure exactly like Arm A would have anyway. Arm A solves it correctly on the first try, no recovery needed.

Where to go next

Two directions, sweep first

01 · try first
Sweep λ

Sweep orchestrator.algo.alpha in configs/arm_b_rl.toml (currently 0.1) upward, or reweight the auxiliary CE loss toward the rare/hard classes instead of uniform per-token weighting. Tells you whether the ASSERTION_FAILURE collapse is a loss-weight problem or a harder generalization ceiling.

02 · then
Reward shaping

Final task reward pays out identically regardless of whether the prediction was right, so GRPO itself has no gradient toward good decisions. Hypothesis: prediction improves decisions, so give extra reward for true failure + predicted failure + REVISE and true PASS + predicted PASS + KEEP — puts the discrimination skill directly in GRPO's reward, not just the auxiliary loss.
