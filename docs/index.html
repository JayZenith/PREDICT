PREDICT: Giving a Coding Agent a World Model It Must Act On

What I'm really after: a coding agent with a working world model, one that doesn't just react to test output but predicts what the environment will do and stakes a decision on that prediction, the way we judge whether our own decisions are worth keeping before acting on them. ECHO showed prediction as a pure training signal helps; PREDICT takes the next step and makes the prediction causally gate behavior: Arm B must commit to KEEP or REVISE before it ever sees the real result. Two questions carry the design. Does a prediction need its own direct supervision, a cross-entropy loss straight off the verified outcome, because RLVR's reward is far too crude to grade one? And is a gate only as wise as its predictor, so calibration has to come before metacognition pays? (A side curiosity rides along: does predicting first sharpen the model's internal state even with no visible tokens to reason in?) Arm A, the matched comparator, is the standard loop: patch, test, react, trained with plain SFT then RLVR. This is early-stage: the mechanism is built and replicated across seeds; whether the gate itself earns its keep is exactly what the next experiments below are designed to answer.

Architecture

One harness, two agents

Both arms share one harness. Arm A reacts to python_test output; Arm B must predict the outcome and commit to KEEP or REVISE before it's allowed to see the real result. The verified-label auxiliary CE design was inspired by Shrivastava, Kauffmann, Awadallah & Papailiopoulos, “ECHO: Terminal Agents Learn World Models for Free” (2026).

Scope
Arm A is a matched comparator, not a single-variable ablation. Arm B bundles the predict/decide protocol, an added action space, the auxiliary CE loss, and a different SFT trace format together: this compares two complete systems, it does not isolate “prediction” as one causal factor. Full caveat in research_specs.md § Limits to report.

Diagnosis

What was actually hindering Arm B

Decision-following vs. discriminationDecision-following is solid: REVISE follows a non-PASS prediction 100% of the time, KEEP follows a PASS prediction 96.4% of the time (seed 42) / 99.4% of the time (seed 43). That's not the problem. The problem is discrimination: telling, from the candidate code alone, before it ever runs, whether it will pass, crash (RUNTIME_ERROR), or run fine but fail the assertion (ASSERTION_FAILURE), the harder case, since it requires simulating the candidate's logic against the test, not just spotting a surface-level crash pattern. Measured by outcome class, RL step 100, seed 42:

Prediction recall by outcome class

Recall = how often the model's own sampled prediction matched the verified outcome. It has never once correctly predicted ASSERTION_FAILURE, at RL step 50, 75, or 100, in either seed, despite it being the single most common verified outcome (57%).

Blind spot
100% PASS at RL step 0 → a brief 1–2% ASSERTION_FAILURE recall at step 25 → extinguished back to 0% by step 50, never recovering through step 100.
Why GRPO can't fix it directly
Reward only depends on whether the final test passes, never on what <PREDICTION> said, so GRPO has no direct loss term on those tokens; the auxiliary CE (λ=0.1) is the only direct teacher for that skill. Shared weights still let GRPO's gradient nudge those tokens indirectly, just weakly.

Root cause: the SFT curriculumPart of why it starts collapsed: half of Arm B's SFT recovery traces hardcode the label PASS on a buggy candidate and let the real test catch the mistake, a genuinely valid recovery pattern, but also exactly the shortcut the RL policy falls back on. Not reward hacking; just indifference, since nothing directly penalizes it. The two experiments below target this: reweighting the CE loss toward rare classes, and reward-shaping GRPO to finally grade the two kinds of rollouts differently.

What this shows
Arm B predicts PASS twice in a row and is wrong both times (the real test comes back ASSERTION_FAILURE), so it gains nothing over Arm A: it reacts to the failure the same way either agent would. Only the third candidate actually passes.
Verifier gap
That third candidate, z = x * y, uses the exact operator the task banned. The test only checks numeric output, not which operator produced it, so it still scores reward 1.0. Arm A's own solution (a loop with sign correction) actually obeys the instruction. Both "solved" the task by the only thing graded; only Arm A solved what was asked. One isolated case in this dataset, not a systemic pattern.

What I learned

Four things this run actually taught

Not the results table: the methodology and architecture lessons that came from digging into why the results looked the way they did.

  1. Reward shapingHalf the training examples quietly taught the model to skip predicting honestly, and reward never punished it. In half of Arm B's recovery examples, the "correct" move was to guess PASS on a buggy candidate and let the real test catch the mistake afterward, a legitimate fallback to teach. But reward pays out the same either way, honest prediction or lazy guess, so the model had no reason to prefer the harder, honest path: it learned the shortcut because nothing in the reward made the shortcut cost anything.
  2. Exposure isn't learningRare at one stage isn't rare at another. Only 37 of 302 SFT labels are real ASSERTION_FAILURE examples, but RL logs show the model sees this outcome constantly during training. The 0% recall isn't a data-scarcity problem.
  3. Noise vs. real winsOne training run is an anecdote. Seed 1 made Arm A look ahead at steps 25 and 100. Both effects were noise that vanished under a second independent run.
  4. Tokens aren't required to thinkPredicting with no visible tokens isn't automatically a dead end. ECHO (this project's inspiration) trains a CE loss with no visible reasoning either, and still doubles pass@1 elsewhere. The likely difference: its target is a dense, multi-token observation; ours is one label from a 6-way enum, which may be why simulation never took hold.

Where to go next

Cheap experiments first, then the harder tests

01 · try first
Turn up λ

λ is the weight on the auxiliary CE loss, currently 0.1. So the only thing that directly trains the prediction is also the smallest term in the loss. Turn it up, and weight the rare failure classes above PASS, then see whether prediction quality moves at all. Cheapest experiment here, and it separates “the signal was too weak” from “a 4B model can't do this.”

02 · try first
Train longer than 100 steps

100 GRPO steps may simply be too short to judge this. Nothing has plateaued: two of the four runs are still climbing at step 100 (+2.8 and +2.4 points over step 75), and the other two are down only 0.6, inside the noise. Comparing arms at a point where both are still moving risks reading a transient as a result. Run to convergence, then compare.

03 · then
Reward shaping

Reward pays out the same regardless of whether the prediction was right, so GRPO has no incentive toward good predictions. Hypothesis: extra reward for correctly-predicted REVISE/KEEP pairs. Caveat: label tokens are still masked from GRPO's own loss, so this reward lands on the surrounding tokens, not the label itself, an indirect nudge, not a direct fix.

04 · the real test
Ablate the gate

The one piece here that's actually new relative to ECHO is that the prediction causally gates KEEP/REVISE. That's never been tested in isolation: run prediction-as-pure-auxiliary-signal (ECHO-style, no behavioral control) against current Arm B, holding everything else fixed.

05 · harder
Denser prediction target

Not another SFT pass on the same label: change what's being predicted. Today's target is a single token from a 6-way enum. Predicting the specific failing assertion or expected/actual value instead forces multi-token, generative simulation of the test, closer to how ECHO's dense observation-token target works, and may be why the model never learned to simulate in the first place.

06 · rigor check
Isolate the causal effect

Right now the prediction and the decision always agree: KEEP follows PASS, REVISE follows a predicted failure, ~100% of the time. So when a rollout goes well, there is no way to tell whether it went well because the prediction was right, or just because of the action that prediction triggered. Break the link: on some rollouts, pick KEEP or REVISE at random instead of following the prediction. If reward still tracks whether the prediction was correct, the prediction is doing real work.

07 · generalize
Scale beyond one model, one benchmark

n=500 on MBPP with a 4B model that may have seen MBPP in pretraining can't rule out idiosyncrasy. Replicating across model scale, or on a benchmark unlikely to be memorized, is what would make this a claim rather than a run.

Appendix

Beyond Code Verification: Research Judgment

One unexpected lesson came from reviewing the experiment itself. I initially described Arm A as a baseline and Arm B as a single added variable. That framing was too clean: Arm B also changed the trace format, action protocol, auxiliary loss, and decision space. The two arms are better understood as comparative training systems, not a strict causal ablation.

The coding agent helped build and document the project, but it did not challenge that experimental claim. Code has hard verifiers: tests, syntax, execution, and logs. Research judgment has no equivalent oracle. Auditing a claim requires reconstructing the entire experiment, identifying every differing factor, and testing whether the conclusion is actually supported.

A stronger research agent would need to do more than make a project coherent and functional. It would need to actively falsify the researcher's framing.
